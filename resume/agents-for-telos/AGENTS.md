# AGENTS.md  
  
## 约束  
- 使用中文沟通  
  
## 项目信息  
- 数据库信息: docs/database/README.md  
- 业务逻辑: docs/business_logic/README.md  
- 问题排查: docs/troubleshooting/README.md  
  
## 协作规则
- 下次开启新任务时，先看本文件，不需要遍历所有 `README.md` 来判断协作方式。  
- 本文件定义了“看哪些文档、何时更新哪个目录、各目录写什么”，是文档协作的入口说明。  
  
## 文档目录边界  
- `docs/business_logic/`：只沉淀稳定业务逻辑（场景、触发条件、状态流转、分支规则、关键代码范围）。  
- `docs/troubleshooting/`：只沉淀问题排查方法（现象、根因、日志、SQL、RCA 过程）。  
- `docs/database/`：数据库结构与表信息说明。  
- `docs/debug.log`：程序运行时 DEBUG 级别日志，如果对话中未提及，该文件中日志与本次任务不相关，不需要阅读。  
  
## 任务执行时的阅读顺序
1. 先读本文件（`AGENTS.md`）。  
2. 若任务涉及业务规则或链路：读 `docs/business_logic/README.md` + 相关主题文档。  
   - 不能只读最相关的一篇文档；要先在 `docs/business_logic/README.md` 中定位业务主题，再按业务对象、入口动作、退出动作、状态字段扩展阅读所有关联文档  
   - 若任务改动的是已有业务链路中的一个局部动作，需要同时阅读该动作的上下游相关文档、以及跟该业务对象/状态相关的文档，确保不会影响原有正常业务逻辑  
3. 若任务是故障定位或线上异常：额外读 `docs/troubleshooting/README.md` 与对应排查文档。  
4. 若任务涉及表结构或字段：读 `docs/database/README.md` 与 `docs/database/tables/`。  
  
## 每次任务后的文档维护要求
- 只要新增/修改了业务规则，必须同步更新 `docs/business_logic/` 对应文档。  
- 只要沉淀了可复用排查方法，必须同步更新 `docs/troubleshooting/` 对应文档。  
- 新增文档时，必须更新对应目录的 `README.md` 索引。  
- 文档只保留当前生效结论，删除或改写过期结论，避免新旧并存。  
- 若本次任务执行过程中因为关联业务文档缺失而导致出现了影响已有正常业务逻辑的错误，需要向用户提出补充文档的要求。  
  
## 代码协作约定  
- 避免提交“无意义代码”。  
- 不要为了 `log.debug` 再包一层 `if (log.isDebugEnabled())`；除非该判断内存在必要的重计算或显著开销逻辑。  
- 在涉及到字符串拼接时，禁止使用`+`、`concat()`等方法，统一使用`""" """.formatted()`，并且要在注释中给出完整的字符串示例  
- 多行注释中的空行需要使用 <p> 占位  
- 如果有注解，注释需要添加在注解上方  
- 注释每行不需要添加末尾标点
- 以下类型的新增代码需要添加注释：  
  * 枚举：含义  
  * 常量：含义  
  * 自动注入的 bean：含义  
  * 全局变量：含义、效果  
  * 方法：功能、应该被使用的场景说明  
- 运行 Maven 命令时串行执行，不要并行执行
